```bash
nvm use                       # picks Node 20 from .nvmrc
npm install
cp .env.example .env          # then fill in MONGODB_URI + JWT_SECRET at minimum
npm run build                 # blog SSG + vite build
npm run dev:full              # backend :5000 + frontend :5173 in parallel
```

If anything in the setup is unclear, the README is wrong — please open a PR fixing it. Don't suffer in silence.

## Branching

- `main` is the only long-lived branch. Vercel auto-deploys from it (production + preview rebuilt on every push).
- Feature branches: `feat/<short-slug>`, `fix/<short-slug>`, `chore/<short-slug>`, `docs/<short-slug>`.
- Open a Pull Request against `main`. Squash on merge if the branch is small (< 4 commits) or merge as-is if each commit tells its own story.
- Never force-push to `main`. Force-push to your own feature branch is fine.

## Commit messages

We follow **Conventional Commits**. The prefix is mandatory:

```
<type>(<scope>): <imperative summary, lowercase, no trailing period>

<optional body>

<optional footer>
```

Types we use:

| Type     | When |
|----------|------|
| `feat`   | New user-facing functionality. |
| `fix`    | Bug fix. |
| `refactor` | Behaviour-preserving code change. |
| `chore`  | Tooling, dependencies, config, repo housekeeping. |
| `docs`   | Documentation only (Markdown, JSDoc, comments). |
| `test`   | Adding or refactoring tests. |
| `content` | Editorial / blog content. |
| `perf`   | Performance fix that isn't a bug. |

Scope is the area touched: `stripe`, `cors`, `commissions`, `tracking`, `onboarding`, `admin`, `chat`, `repo`, `frontend`, `deps`, `lint`, etc. Keep it short.

Body explains the **why** (not the what — the diff already shows the what). Reference the audit item if applicable: `Refs: AUDIT.md A-7`.

Look at recent commits for style: `git log --oneline -20`.

### Co-authoring

If a commit is generated by Claude Code (or any AI assistant) on your machine, append a `Co-Authored-By:` trailer so reviewers know:

```
Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

For commits you wrote yourself, no trailer needed — Git's own author field is enough.

## Pull Requests

PR title = the conventional-commit summary of the squash result. Body should answer:

1. **What** changes (1-3 bullets).
2. **Why** — the user problem, the bug report, the audit item.
3. **How to verify** — the smoke test, the affected route, the env var that needs to be set.
4. **Risk** — what could break, and what the rollback is.
5. **Screenshots** for visible UI changes.

If your PR closes an audit item, mention `Refs: AUDIT.md X-N` in the body so the next review of `AUDIT.md` can move it to "resolved".

CI must be green before merge:
- `npm run lint` (0 errors; warnings allowed).
- `npm test` (jest + supertest).
- `npm run build` (blog SSG + vite).

## Code style

- **Language:** existing modules are bilingual (Spanish for legacy domain entities — `Canal`, `Anuncio`, `Usuario`, `Transaccion` — and English for newer modules and infra). When you add **new** modules, prefer English. When you edit existing Spanish modules, keep their language consistent. The [glossary](docs/glossary.md) maps the two.
- **Indentation:** 2 spaces, LF line endings. (Windows users: git auto-converts to CRLF on checkout, that's fine.)
- **Strings:** single quotes in JS, double in JSX attributes.
- **Imports:** group by source — node built-ins, third-party, then local; one blank line between groups.
- **Comments:** explain *why*, not *what*. The diff and well-named identifiers cover *what*. Don't reference the current task or PR ("added for issue #X") — that rots; put it in the PR description.
- **No `console.log`/`console.error` in new server code.** Use `lib/logger` (Winston). The existing `console.error('FOO ERROR:', e)` calls in older auth code are an active migration (see audit A-11).

## Testing

- Unit tests live next to no source — they're all under `tests/` in a flat layout (`*.test.js` for unit, `*.integration.test.js` for HTTP tests via supertest).
- Use the existing `tests/helpers/` setup for DB fixtures.
- Integration tests need `MONGODB_TEST_URI` set in `.env` (separate database from your dev one).
- New features must come with a test. Bug fixes should land with a regression test that fails on the parent commit.

## Money & secrets

These rules are non-negotiable:

- **Every Stripe API call must pass an `idempotencyKey`** (see existing pattern: `transfer:<campaignId>`, `pi:<txId>`, etc.). Without it, a network retry double-charges or double-pays.
- **Webhook handlers must verify signatures.** Stripe uses `stripe.webhooks.constructEvent(req.body, sig, secret)` with the raw body. Meta uses HMAC-SHA256 of the raw body with `META_APP_SECRET`. See `routes/partnerWebhook.js` and `routes/webhooks.js` for the canonical patterns.
- **Commission rates live in `config/commissions.js`.** Never hardcode a percentage anywhere else; import from there.
- **No secret in code or in a committed file.** `.env` is gitignored; production secrets live in Vercel project settings. If you suspect a leak, rotate per [SECURITY.md](SECURITY.md) and assume it's compromised.

## Where to look first (your first hour)

1. [README.md](README.md) — mental model.
2. [docs/architecture.md](docs/architecture.md) — the 6 main pieces.
3. [docs/glossary.md](docs/glossary.md) — so `Canal` and `Anuncio` stop being puzzles.
4. `app.js` — the Express setup, including the Vercel `_routes`/`safeMount` pattern.
5. `models/Campaign.js`, `models/Canal.js`, `models/Usuario.js` — the three schemas behind every flow.
6. `controllers/campaignController.js` — the campaign lifecycle (DRAFT → PAID → PUBLISHED → COMPLETED), where escrow capture and creator payout happen.
7. `client/src/routes/AppRoutes.jsx` — the React route map.

Then pick a `good first issue` from GitHub or ask in #engineering for something starter-sized.
